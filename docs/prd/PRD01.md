## 1. Purpose

هدف این سند تعریف نسخه اول محصول Sales Health Check است.

این نسخه قرار نیست کل سیستم EMC، کل دوره قیف فروش یا یک پلتفرم کامل مدیریتی باشد.

هدف MVP این است که یک مدیر SME بتواند وضعیت مسیر فروش خود را ارزیابی کند، گلوگاه‌های اصلی را ببیند و برای ۷ روز و ۳۰ روز آینده اقدام مشخص دریافت کند.

---

## 2. Product Problem

مدیران SME معمولاً فروش دارند، تبلیغ می‌کنند، محتوا تولید می‌کنند، فروشنده دارند یا با مشتری صحبت می‌کنند؛ اما اغلب نمی‌دانند مشکل اصلی فروش از کجاست.

مشکل ممکن است از یکی از این بخش‌ها باشد:

- مشتری اشتباه
- پیام نامشخص
- پیشنهاد ضعیف
- ورودی کم یا بی‌کیفیت
- پاسخ‌دهی کند
- صلاحیت‌سنجی ضعیف
- مکالمه فروش غیرحرفه‌ای
- ضعف در پیگیری
- تجربه مشتری ناهماهنگ
- نبود عدد و تحلیل

بدون یک ابزار تشخیصی، مدیر معمولاً با حدس تصمیم می‌گیرد.

این محصول قرار است حدس را به تشخیص تبدیل کند.

---

## 3. Target User

کاربر اصلی MVP:

مدیر یا صاحب کسب‌وکار کوچک و متوسط که فروش دارد، اما فروشش قابل پیش‌بینی، قابل تحلیل و قابل بهبود نیست.

ویژگی‌ها:

- معمولاً تیم کوچک دارد.
- ممکن است فروشنده، ادمین یا تیم بازاریابی داشته باشد.
- بهبود فروش برایش فوریت دارد.
- وقت زیادی برای تحلیل پیچیده ندارد.
- گزارش ساده، قاطع و قابل اجرا می‌خواهد.

کاربر MVP فعلاً این‌ها نیست:

- شرکت Enterprise
- تیم فروش بزرگ با CRM پیچیده
- مشاور حرفه‌ای فروش
- کاربر عمومی بدون کسب‌وکار فعال

## 8. User Journey

### Step 1 — Landing Page

کاربر وارد صفحه معرفی می‌شود.

باید سریع بفهمد:

- این ابزار چیست؟
- برای چه کسی است؟
- چه خروجی‌ای می‌دهد؟
- چقدر زمان می‌برد؟

CTA اصلی:

شروع ارزیابی فروش

### Step 2 — Basic Information

کاربر اطلاعات اولیه را وارد می‌کند:

- نام
- شماره تماس یا ایمیل
- نام کسب‌وکار
- حوزه فعالیت
- اندازه تقریبی تیم
- مدل فروش: آنلاین، حضوری، تلفنی، ترکیبی

### Step 3 — Health Check Form

کاربر به سوالات Health Check پاسخ می‌دهد، اما منطق پاسخ‌ها نباید کیفی، احساسی یا سلیقه‌ای باشد.

اصل طراحی فرم:

کاربر نباید مجبور شود بگوید «من خوبم یا ضعیفم».

کاربر باید چیزی را انتخاب کند که واقعاً در کسب‌وکارش می‌بیند.

بنابراین گزینه‌ها باید Evidence-Based باشند؛ یعنی بر اساس شواهد، رفتارها، فرایندها، عددها یا وضعیت‌های قابل مشاهده طراحی شوند.

مثال بد:

سؤال: آیا قیف فروش شما شفاف است؟

گزینه‌ها:

- اصلاً
- کمی
- نسبتاً
- کاملاً

این مدل مناسب نیست، چون کاربر را مجبور به قضاوت کیفی می‌کند.

مثال درست:

سؤال: وقتی یک مشتری جدید با شما تماس می‌گیرد، معمولاً می‌دانید اولین بار از کجا با شما آشنا شده است؟

گزینه‌ها:

- 0 = معمولاً نمی‌دانیم و جایی ثبت نمی‌شود.
- 1 = گاهی از مشتری می‌پرسیم، اما منظم ثبت نمی‌شود.
- 2 = بیشتر اوقات منبع آشنایی را ثبت می‌کنیم، اما تحلیل منظمی نداریم.
- 3 = منبع آشنایی تقریباً همیشه ثبت و منظم بررسی می‌شود.

بنابراین در MVP، هر سؤال باید این ویژگی‌ها را داشته باشد:

1. درباره یک رفتار یا واقعیت مشخص در کسب‌وکار باشد.
2. گزینه‌ها نماینده چهار سطح بلوغ واقعی باشند.
3. کاربر بتواند بدون دانش تخصصی گزینه درست را انتخاب کند.
4. گزینه‌ها به جای حس و برداشت، بر اساس شواهد قابل مشاهده باشند.
5. هر گزینه مستقیماً به امتیاز تشخیصی وصل شود.

### Step 4 — Calculation

سیستم امتیاز هر دامنه را بر اساس گزینه‌های انتخاب‌شده محاسبه می‌کند.

برای هر سؤال، گزینه‌ها از ۰ تا ۳ امتیاز دارند.

اما این امتیازها نماینده «کیفیت ذهنی پاسخ» نیستند؛ نماینده سطح بلوغ مشاهده‌شده در کسب‌وکار هستند.

برای هر دامنه:

Domain Raw Score = مجموع امتیاز سوالات همان دامنه

Domain Max Score = تعداد سوالات دامنه × ۳

Domain Percentage = Domain Raw Score / Domain Max Score × 100

در نسخه MVP، هر دامنه می‌تواند ۳ تا ۵ سوال داشته باشد، اما اگر تعداد سوالات دامنه‌ها برابر نبود، برای نمایش دامنه‌ای از درصد استفاده می‌شود.

با این حال، امتیاز کلی محصول با عدد کل سنجیده می‌شود، نه میانگین ساده درصد دامنه‌ها.

### Step 5 — Result Page

کاربر صفحه نتیجه را می‌بیند:

- امتیاز کلی
- وضعیت کلی
- Spider Chart
- امتیاز ۱۶ دامنه
- وضعیت ۴ لایه
- ۳ گلوگاه اصلی
- گزارش کوتاه تشخیصی
- اقدام ۷ روزه
- اقدام ۳۰ روزه

## 9. Scoring Logic

### Question Score

هر سؤال از ۰ تا ۳ امتیاز دارد.

اما گزینه‌ها نباید به شکل «ضعیف، متوسط، خوب» نوشته شوند.

گزینه‌ها باید چهار سطح واقعی و قابل مشاهده از بلوغ همان رفتار را نشان دهند.

نمونه الگوی گزینه‌ها:

- 0 = این رفتار/فرایند وجود ندارد یا کاملاً اتفاقی است.
- 1 = گاهی انجام می‌شود، اما فردمحور و نامنظم است.
- 2 = تا حدی انجام می‌شود، اما هنوز کامل، مستند یا قابل تحلیل نیست.
- 3 = منظم، قابل تکرار، ثبت‌شده و قابل بررسی است.

این الگو باید برای هر سؤال با زبان همان سؤال بازنویسی شود، نه اینکه به صورت عمومی در فرم تکرار شود.

### Domain Raw Score

هر دامنه از مجموع امتیاز سؤال‌های همان دامنه ساخته می‌شود.

Domain Raw Score = مجموع امتیاز سؤال‌های دامنه

Domain Max Score = تعداد سؤال‌های دامنه × ۳

Domain Percentage = Domain Raw Score / Domain Max Score × 100

Domain Percentage برای نمایش وضعیت دامنه، Spider Chart و مقایسه بین دامنه‌ها استفاده می‌شود.

### Domain Health Level

- 0 تا 25 درصد: بحرانی
- 26 تا 50 درصد: ضعیف
- 51 تا 75 درصد: متوسط
- 76 تا 100 درصد: سالم

### Overall Score

امتیاز کلی نباید میانگین ساده درصد ۱۶ دامنه باشد.

امتیاز کلی باید با عدد کل سنجیده شود.

Overall Raw Score = مجموع امتیاز همه سؤال‌ها

Overall Max Score = مجموع حداکثر امتیاز همه سؤال‌ها

Overall Score = Overall Raw Score / Overall Max Score × 100

دلیل این تصمیم:

اگر تعداد سؤال‌ها در دامنه‌ها برابر نباشد، میانگین ساده دامنه‌ها می‌تواند تصویر غلط بدهد.

امتیاز کلی باید نشان دهد کاربر از کل ظرفیت Health Check چند امتیاز گرفته است.

### Weighted Bottleneck Detection

> **Addendum (2026-06-27):** Diagnosis priority for new assessments uses EMC Engine v2 (PI, survival gate, question-level roots) when `ModelVersion.diagnosisEngineVersion = "v2"`. The formula below remains **Legacy v1** for backward compatibility. Overall Score rule unchanged (ADR 0002).

تشخیص گلوگاه نباید فقط بر اساس پایین‌ترین درصد دامنه‌ها باشد.

چون همه ضعف‌ها اثر یکسان روی فروش ندارند.

مثلاً ضعف در Persona، UVP یا Offer می‌تواند کل مسیر فروش را خراب کند، حتی اگر چند دامنه اجرایی امتیاز متوسط بگیرند.

بنابراین Bottleneck باید با وزن‌دهی تشخیص داده شود.

برای MVP، هر دامنه یک وزن اثرگذاری دارد.

پیشنهاد اولیه وزن‌ها:

Layer 1 — بازار، پیام و پیشنهاد

- Persona: وزن 1.4
- UVP: وزن 1.4
- Offer Design: وزن 1.3

Layer 2 — لید و مدیریت سرنخ

- Lead Generation: وزن 1.2
- Lead Nurturing: وزن 1.1
- Speed to Lead: وزن 1.2
- Lead Qualification: وزن 1.2

Layer 3 — مکالمه فروش و تبدیل

- Initial Trust: وزن 1.1
- Needs Discovery: وزن 1.3
- Presentation: وزن 1.1
- Objection Handling: وزن 1.2
- Closing: وزن 1.2

Layer 4 — رابطه، تجربه و بهینه‌سازی

- Loyalty: وزن 1.0
- Touchpoint Consistency: وزن 1.0
- Sales Journey Clarity: وزن 1.3
- Measurement & Optimization: وزن 1.3

### Bottleneck Priority Score

برای تشخیص گلوگاه، سیستم باید شدت ضعف را در وزن اثرگذاری ضرب کند.

Weakness Score = 100 - Domain Percentage

Bottleneck Priority Score = Weakness Score × Domain Weight

سه دامنه‌ای که بالاترین Bottleneck Priority Score را دارند، گلوگاه‌های اصلی گزارش می‌شوند.

این یعنی ممکن است دامنه‌ای با درصد کمی بهتر، اما وزن اثرگذاری بالاتر، نسبت به دامنه‌ای با درصد پایین‌تر اولویت بیشتری بگیرد.

### Important Rule

در MVP، وزن‌ها باید در کد یا دیتابیس قابل تنظیم باشند، نه اینکه در منطق برنامه hard-code غیرقابل تغییر شوند.

چون بعد از تست با کاربران واقعی، ممکن است وزن‌ها اصلاح شوند.

## 10. Report Logic

گزارش کاربر باید این بخش‌ها را داشته باشد:

### 1. وضعیت کلی

مثال:

فروش شما در وضعیت متوسط قرار دارد. یعنی پایه‌هایی از مسیر فروش وجود دارد، اما هنوز چند گلوگاه جدی باعث می‌شود بخشی از فرصت‌های فروش از دست برود.

### 2. گلوگاه‌های اصلی

برای هر گلوگاه:

- نام دامنه
- وضعیت فعلی
- معنی ضعف در این دامنه
- اثر مستقیم روی فروش
- اقدام پیشنهادی

### 3. تحلیل لایه‌ای

هر ۴ لایه باید وضعیت کوتاه داشته باشد:

- سالم
- قابل بهبود
- ضعیف
- بحرانی

### 4. اقدام ۷ روزه

اقدام‌های فوری، ساده و کم‌هزینه.

مثلاً:

- بررسی ۳۰ لید اخیر
- نوشتن مسیر واقعی مشتری
- ثبت سوالات پرتکرار مشتری
- طراحی یک فرم ساده پیگیری

### 5. اقدام ۳۰ روزه

اقدام‌های ساختاری‌تر.

مثلاً:

- ساخت Playbook پیگیری فروش
- طراحی پیام ارزش پیشنهادی
- ساخت سیستم ثبت Lead Source
- ساخت داشبورد ساده KPI فروش

## 11. Functional Requirements

### FR1 — User can start assessment

کاربر باید بتواند از Landing Page وارد فرم Health Check شود.

### FR2 — User can submit basic business info

سیستم باید اطلاعات پایه کسب‌وکار را ذخیره کند.

### FR3 — User can answer assessment questions

سیستم باید سوالات را نمایش دهد و پاسخ‌ها را دریافت کند.

### FR4 — System calculates domain scores

بعد از ارسال فرم، سیستم باید امتیاز هر دامنه را محاسبه کند.

### FR5 — System generates overall score

سیستم باید امتیاز کلی سلامت فروش را محاسبه کند.

### FR6 — System identifies top bottlenecks

سیستم باید ۳ دامنه ضعیف‌تر را به عنوان گلوگاه اصلی تشخیص دهد.

### FR7 — System displays result dashboard

کاربر باید صفحه نتیجه را ببیند.

### FR8 — System displays spider chart

سیستم باید نمودار راداری / Spider Chart از دامنه‌ها نمایش دهد.

### FR9 — System generates simple diagnostic report

سیستم باید گزارش متنی ساده تولید کند.

### FR10 — User can revisit result page

کاربر باید بتواند بعداً دوباره نتیجه را مشاهده کند.

---

## 12. Non-Functional Requirements

### Simplicity

محصول باید برای مدیر SME ساده و سریع باشد.

### Speed

فرم نباید آنقدر طولانی شود که کاربر رها کند.

هدف زمانی:

۱۰ تا ۱۵ دقیقه برای تکمیل.

### Reliability

امتیازدهی باید قابل پیش‌بینی و قابل تست باشد.

### Maintainability

مدل سوالات، دامنه‌ها و گزارش‌ها باید طوری طراحی شود که بعداً قابل تغییر باشد.

### AI-Native Readiness

حتی اگر MVP ابتدا با Rule-Based Report شروع شود، معماری باید آماده اضافه‌کردن AI Analysis باشد.

---

## 13. Success Metrics

MVP زمانی موفق است که:

1. کاربر بدون توضیح دستی بتواند فرم را تکمیل کند.
2. حداقل ۶۰ درصد کاربران شروع‌کننده، فرم را کامل کنند.
3. کاربر بعد از دیدن گزارش بگوید مشکل اصلی فروشش روشن‌تر شده است.
4. گزارش بتواند برای فروش دوره، مشاوره یا عضویت EMC مکالمه ایجاد کند.
5. سیستم برای حداقل ۳۰ کاربر واقعی بدون خطای جدی اجرا شود.

---

## 14. Definition of Done

MVP زمانی Done است که:

- Landing Page آماده باشد.
